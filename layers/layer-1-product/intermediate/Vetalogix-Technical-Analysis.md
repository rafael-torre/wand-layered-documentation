# VETALOGIX
## Technical Analysis & Open Questions

---

## System Overview

The client is proposing a veterinary symptom-checker with a secure, multi-tier AI pipeline hosted on Google Cloud Platform, plus an admin layer for the owner to export data and update AI logic.

The core user flow is:

- A dog owner selects symptoms from a structured form (355 symptoms across grouped categories) along with their dog's profile (breed, age, weight, sex, medications, vaccination history, and lifestyle data).
- The symptom input is run through a deterministic scoring engine — the Valtorta Logic Engine — that ranks conditions by a weighted diagnostic score. An LLM layer then generates a plain-language narrative for the top results.
- The system returns a ranked triage output to the user's dashboard, with urgency tiering (Critical / Urgent / Monitor) and a plain-language Diagnostic Handshake for each result.
- All case data is archived anonymously. The platform owner can export reports and push logic updates from a private admin console.

### Layer 0 — User Interface
*Client name: ETHOS UI / Structured Input*

**What it is**

A structured web form where a dog owner submits their dog's profile and selects symptoms. All input is constrained to dropdowns — no free-text entry. The output is a structured JSON payload sent downstream to the scoring engine. The form has two main sections: the K9 Ethos Profile and the symptom selector.

The K9 Ethos Profile captures:
- **BIOMETRICS:** Name, Breed, Spayed/Neutered, Gender, Age, Height, Weight
- **LIFESTYLE:** Activity Level, Food type, Daily Walks, Sleep Schedule
- **HEALTH:** Up to 6 current medications, up to 6 prior diagnoses with dates
- **VACCINATIONS:** Up to 6 vaccines with dates received

The symptom selector captures:
- 1 to 3 Primary Symptoms (each gets a 1.5x scoring multiplier in the engine)
- 1 to 10 Secondary Symptoms (each gets a 1.0x multiplier)
- Each symptom entry has: a category dropdown, a specific symptom dropdown, and a duration field
- Known symptom categories from test data: Movement & Energy, Vital Functions, Physical & Surface, Blood & Urgency
- Five Sentry binary fields (YES/NO): Appetite status, Activity/Energy status, Gum color (pink and moist?), Stool quality (solid?), Temperature (feels warm?)
- Contextual fields: Recent Travel (yes/no), Primary Region

**Objectives**

- Present grouped dropdown menus covering 355 clinical symptoms across categorized groups.
- Capture the full K9 Ethos Profile (biometrics, lifestyle, health history, medications, vaccinations) as described above.
- Allow selection of 1–3 Primary Symptoms and 1–10 Secondary Symptoms, each with category, specific symptom, and duration.
- Capture the 5 Sentry binary fields and contextual fields (Recent Travel, Primary Region).
- Package all input into a Patient_Profile JSON object and a Diagnostic_Input JSON object for handoff downstream.
- Strip owner PII at submission — only clinical and dog data moves forward.
- Display the Valtorta Response on the user's Ethos Dashboard: ranked conditions with SGS scores, urgency tier color coding, and Diagnostic Handshake narratives.

**Open Questions**

- Is this a standalone web app, or is it embedded in the existing WANDVERSE platform / WordPress site?
- What does the Ethos Dashboard show beyond the current case response — is there case history, saved profiles for multiple dogs, trends over time?

### Level 2 — Data Ingestion & Sanitization Layer
*Client name: The Landing Zone / The Dirt*

**What it is**

A data staging layer in BigQuery and Cloud Storage that receives the Sovereign Case Object from Level 1, verifies PII has been stripped, validates the data structure, and transforms it into the two JSON objects expected by the Level 3 scoring engine. This layer sits behind a VPC boundary with no direct access from the public-facing Level 1.

The **input** to this layer is the full Sovereign Case Object — the rich payload originating from the UI, which includes the complete K9 Ethos Profile (biometrics, lifestyle, health history, medications, vaccinations), all selected symptoms with categories and durations, the five Sentry fields, contextual fields (region, travel), and the Level 1 triage summary.

The **output** of this layer — what gets passed to Level 3 — is a leaner, sanitized structure derived from the test data and code schema:

```
Patient_Profile: { breed, age, sex, medical_history[], sentries { appetite, energy, gums, stool } }
Diagnostic_Input: { primary_symptoms[], secondary_symptoms[], duration }
```

The sentry fields (appetite, energy, gums, stool) are the binary YES/NO stability markers captured in the form. These are the only "biometric" data in the system — there is no hardware sensor or biometric device component.

Note that the UI collects significantly more fields than what currently appears in the output schema (height, weight, lifestyle, vaccinations, region, travel). Whether those fields are intentionally excluded from scoring, need to be added to the schema, or flow only to the archive (Level 4) is an open question.

**Objectives**

- Receive the Sovereign Case Object via a secure internal channel (Pub/Sub or equivalent; not a direct API call).
- Verify PII redaction: confirm no identifying owner data is present.
- Transform the full Ethos Profile payload into the Patient_Profile and Diagnostic_Input JSON structures required by the Level 3 scoring engine.
- Store the complete raw case object in BigQuery / Cloud Storage for audit and archive purposes.
**Open Questions**

- The UI collects height, weight, vaccination history, lifestyle data, region, and recent travel — none of which appear in the current Patient_Profile schema for the scoring engine. Are these fields intentionally excluded from scoring, or do they need to be incorporated? If they're excluded, are they still passed to Level 4 for archiving?
- What does the 'one-way bridge' mean in practice? Is this a message queue (e.g., Pub/Sub) so Level 1 cannot read back from Level 2, or something else?

### Level 3 — Core AI Processing Engine
*Client name: The Sovereign Intel Vault / AI #2 (The Logos / Gemma Agent)*

**What it is**

The primary diagnostic engine. It receives the sanitized Patient_Profile and Diagnostic_Input from Level 2 and runs them through the Valtorta Logic Engine — a deterministic weighted-scoring algorithm — against the Master IDS (397 conditions). The top-ranked conditions are then passed to an LLM (Gemma) which generates the plain-language Diagnostic Handshake narrative. The final output (the Valtorta Response) is returned to the user's dashboard and archived to Level 4.

**The Valtorta Logic Engine — Total Diagnostic Score (TDS) Formula**

For each of the 397 conditions in the Master IDS, the engine calculates:

```
TDS = (Symptom_Vector × Age_Factor) + Breed_Surge + History_Surge + Acute_Surge − Stability_Penalty + Prevalence_Headwind
```

The six scoring components are:

**1. Anatomical Gate (Fg)**
If a condition has a gender requirement that doesn't match the patient, it receives a score of -100 and is excluded. This is a hard gate, not a soft penalty.

**2. Symptom Velocity Engine (Σ W × M)**
Each submitted symptom is matched against the condition's weight in the IDS. Primary symptoms receive a 1.5x multiplier; Secondary symptoms receive 1.0x. Weights can be negative (e.g., -2.0 for Hard Abdomen in Heat Stroke) to enable mutual exclusion logic.

**3. Contextual Surges**
- Age Factor (Af): 1.5x for geriatric conditions if patient age > 8; 0.5x for pediatric if age < 1; 1.0x otherwise.
- Breed Surge (Bm): +1.0 if the patient's breed is in the condition's high-risk breed list.
- History Surge (Hh): +2.0 if the condition appears in the patient's known medical history.
- Acute Emergency Surge (Ea): +0.5 for crisis-type conditions when duration is Seconds/Minutes.

**4. Sentry Stability Brakes (Ss)**
For high-lethality conditions, each stable sentry marker (Appetite: YES, Energy: YES, Gums: YES, Stool: YES) applies a -2.0 penalty. A dog showing normal appetite and energy is unlikely to be in an acute high-lethality crisis — this is the mechanism that suppresses toxicity and hemorrhage diagnoses when stability markers are positive.

**5. Prevalence Headwinds (Ph)**
A penalty of 0.0, -0.25, or -1.0 applied to rare conditions to prevent statistical outliers from dominating results.

**6. Result Normalization**
All 397 TDS scores are sorted descending. The Top 10 are surfaced.

**The Valtorta Response Output Format**

Results are bucketed into three urgency tiers based on the Sovereign Grip Score (SGS):

| Zone | SGS Range | Action Command | UI Color |
| :---- | :---- | :---- | :---- |
| **CRITICAL STRIKE** | 3.0 + | EMERGENCY: GO NOW | Neon Red |
| **URGENT CONSULT** | 1.5 – 2.9 | CLINICAL REVIEW REQUIRED | Amber Yellow |
| **MONITOR / GUIDANCE** | Below 1.5 | OBSERVE & DOCUMENT | Cobalt Blue |

Each result includes the condition name, SGS score, and a Diagnostic Handshake narrative (LLM-generated) explaining why the condition matched.

**Objectives**

- Receive the Patient_Profile and Diagnostic_Input JSON from Level 2.
- Execute the TDS formula against all 397 conditions in the Master IDS.
- Apply all six scoring components: Anatomical Gate, Symptom Velocity, Contextual Surges, Sentry Brakes, Prevalence Headwinds, Result Normalization.
- Sort results, extract the Top 10, and assign urgency tier and color.
- Generate a Diagnostic Handshake narrative for each top result (LLM layer).
- Return the Valtorta Response to the user's dashboard.
- Pass the full case record (input + scores + response) to Level 4 for archiving.
- Run entirely within a locked VPC perimeter.

**Open Questions**

- The Master IDS with 397 conditions is the core data asset. What format is it in today (spreadsheet, CSV, database)? We need access to the full dataset — including symptom weights, breed risk flags, lethality classifications, and prevalence headwinds for all 397 conditions — before Level 3 can be scoped or built.
- Who is responsible for validating that the symptom weights and condition classifications in the IDS are clinically accurate? Is there a licensed veterinarian involved in calibrating and signing off on the dataset?
- The current Prevalence Headwinds penalize rare conditions to reduce noise. The Anomaly Mandate calls for surfacing unusual symptom combinations. These two directives are in tension. How should the engine resolve this — should rare conditions receive a boost when multiple primary symptoms align strongly, overriding the headwind?
- Has any version of the IDS or scoring logic been implemented in code anywhere beyond the spreadsheet, or is the spreadsheet the current source of truth?

## Security Architecture

The client's 'Sanctuary Protocol' / 'Augusta Standard' maps to GCP's VPC Service Controls framework. The high-level intent is sound — tiered isolation with progressively restricted access as data moves deeper. The implementation, however, raises operational concerns.

**What is being asked for**

- Four VPC service perimeters enforced by GCP VPC Service Controls and IAM.
- Levels 3 and 4 wrapped in a 'Black Box' perimeter; engineering accounts have no read/write access post-deployment.
- Only the Gemma Service Account (automated AI pipeline) and the David Executive ID have access to Levels 3 and 4.
- Full audit logging via Cloud Audit Logs on every access attempt to the inner perimeters.

**Open Questions**

- If engineering is air-gapped from Levels 3 and 4 after deployment, who handles production incidents in those layers? What happens when the AI starts returning bad results or a service goes down?
- How do we deploy updates to the Level 3 scoring engine or IDS dataset if engineering has no access? Is there a defined ops model for ongoing maintenance?
- Is the 'air-gap' meant to be permanent (no engineer ever accesses production data) or can access be granted temporarily with audit logging, on a break-glass basis?
- VPC Service Controls create 'logical' perimeters — data can still flow between projects with the right service account permissions. What is the client's definition of 'physically isolated'? We need to align on what this means technically.

## Cross-Cutting Concerns

The following issues apply across multiple layers and need to be resolved before the system can be scoped or designed.

### 1. The Master IDS is the product — and we need access to it.

Every other component in this architecture is infrastructure. The actual value proposition of VETALOGIX lives in the 397-condition Master IDS: its symptom weights, breed risk classifications, lethality levels, and prevalence headwinds. The Valtorta Logic Engine algorithm is now defined and understood. The remaining gap is the dataset itself. Until we have the IDS in a usable format and can confirm the data quality, Level 3 cannot be scoped, estimated, or built. This is the first thing to resolve.

### 2. Regulatory and liability exposure.

The output uses urgency tiers framed as action commands ("EMERGENCY: GO NOW", "CLINICAL REVIEW REQUIRED", "OBSERVE & DOCUMENT"). This is more defensible than a "definitive diagnosis," but AI systems that generate health assessments — even for animals — can attract FDA or USDA scrutiny depending on how they are marketed and used. Before the product is designed, the client needs legal input on how outputs must be framed and disclaimed. This shapes UX copy and disclaimers throughout the system.
