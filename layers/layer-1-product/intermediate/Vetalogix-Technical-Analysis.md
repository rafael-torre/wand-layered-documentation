# VETALOGIX
## Technical Analysis & Open Questions

## System Overview

The client is proposing a veterinary symptom-checker with a secure, multi-tier AI pipeline hosted on Google Cloud Platform, plus an admin layer for the owner to export data and update AI logic.

The core user flow is:

- A dog owner selects symptoms from a structured form (355 symptoms across grouped categories) along with their dog's profile (breed, age, weight, sex, medications, vaccination history, and lifestyle data).
- The symptom input is run through a deterministic scoring engine — the Valtorta Logic Engine — that ranks conditions by a weighted diagnostic score. An LLM layer then generates a plain-language narrative for the top results.
- The system returns a ranked triage output to the user's dashboard, with urgency tiering (Critical / Urgent / Monitor) and a plain-language Diagnostic Handshake for each result.
- All case data is archived anonymously. The platform owner can export reports and push logic updates from a private admin console.

### Layer 0 — User Interface
*Client name: ETHOS UI / Structured Input*

**What it is**

A structured web form where a dog owner submits their dog's profile and selects symptoms. All input is constrained to dropdowns — no free-text entry. The output is a structured JSON payload sent downstream to the scoring engine. The form has two main sections: the K9 Ethos Profile and the symptom selector.

The K9 Ethos Profile captures:
- **BIOMETRICS:** Name, Breed, Spayed/Neutered, Gender, Age, Height, Weight
- **LIFESTYLE:** Activity Level, Food type, Daily Walks, Sleep Schedule
- **HEALTH:** Up to 6 current medications, up to 6 prior diagnoses with dates
- **VACCINATIONS:** Up to 6 vaccines with dates received

The symptom selector captures:
- 1 to 3 Primary Symptoms (each gets a 1.5x scoring multiplier in the engine)
- 1 to 10 Secondary Symptoms (each gets a 1.0x multiplier)
- Each symptom entry has: a category dropdown, a specific symptom dropdown, and a duration field
- Known symptom categories from test data: Movement & Energy, Vital Functions, Physical & Surface, Blood & Urgency
- Five Sentry binary fields (YES/NO): Appetite status, Activity/Energy status, Gum color (pink and moist?), Stool quality (solid?), Temperature (feels warm?)
- Contextual fields: Recent Travel (yes/no), Primary Region

**Objectives**

- Present grouped dropdown menus covering 355 clinical symptoms across categorized groups.
- Capture the full K9 Ethos Profile (biometrics, lifestyle, health history, medications, vaccinations) as described above.
- Allow selection of 1–3 Primary Symptoms and 1–10 Secondary Symptoms, each with category, specific symptom, and duration.
- Capture the 5 Sentry binary fields and contextual fields (Recent Travel, Primary Region).
- Package all input into a Patient_Profile JSON object and a Diagnostic_Input JSON object for handoff downstream.
- Strip owner PII at submission — only clinical and dog data moves forward.
- Display the Valtorta Response on the user's Ethos Dashboard: ranked conditions with SGS scores, urgency tier color coding, and Diagnostic Handshake narratives.

**Deployment Context**

- The UI will be built on the existing **wvere.com platform**, not WordPress. The current website will be rebuilt from scratch to support the diagnostic engine architecture and biometric authentication requirements.
- A marketing company is contracted to handle design, branding, and general website content. Your technical team owns the diagnostic engine implementation, biometric sign-in (D-Scope), and security protocols.
- Users access via a biometric login gate (D-Scope) before being directed to their personal Ethos Dashboard. This gate also prevents unauthorized bot access.

### Level 2 — Core AI Processing Engine
*Client name: The Sovereign Logos Agent / The Intelligence Hub*

**What it is**

This is the Vetalogix "brain" — the single layer where all clinical synthesis occurs. Upon receiving the raw JSON package from Level 1, the AI Agent performs all clinical operations in one place, querying Level 3 (the Resource Library) as needed for reference data.

The **input** to this layer is the Sovereign Case Object — the full Ethos Profile and symptom payload packaged by Level 1.

The **three operations** Level 2 performs:

1. **Statistical Math** — Runs the raw IDS weighting against the submitted symptoms (the Symptom Velocity Engine and all TDS formula components).
2. **Valtorta Multipliers** — Applies the Genetic Surge (B_m) using breed data from the Level 3 Resource Library; calculates Sentry Brakes (Ss) by cross-referencing bio-data against symptoms.
3. **Branded Response & Validation** — Generates the Diagnostic Handshake narrative for each top result, flags Logic Conflicts (e.g., an age-impossible match), and runs a validation check against Merck Manual reference data to ensure the results are clinically plausible. If it does not passed it should be flagged.

The sentry fields (appetite, energy, gums, stool) are the binary YES/NO stability markers captured in the form. These are the only "biometric" data in the system — there is no hardware sensor or biometric device component.

**Objectives**

- Receive the Sovereign Case Object from Level 1 via a secure internal channel (Pub/Sub or equivalent).
- Verify PII redaction: confirm no identifying owner data is present.
- Execute the full TDS formula against all 397 conditions, querying Level 3 for breed, Merck Manual, and condition tag data as needed.
- Apply all scoring components: Anatomical Gate, Symptom Velocity, Contextual Surges (including Medications factor), Sentry Brakes, Prevalence Headwinds, Result Normalization.
- Sort results, extract the Top 10, assign urgency tier and color.
- Generate a Diagnostic Handshake narrative for each top result and flag any Logic Conflicts.
- Return the Valtorta Response to the user's Ethos Dashboard.
- Pass the complete Synthesis Log (raw Dirt + final scores + Valtorta Response) to Level 4 for archiving.

### Level 3 — Resource Library
*Client name: The Sovereign Intel Vault / The Resource Library*

**What it is**

A modular, static repository of hardened CSV data assets. Level 3 holds no processing logic — it is reference data that Level 2 queries as needed during clinical synthesis. Keeping this data separate from the processing engine means clinical weights, breed tables, and condition entries can be updated without touching any application code.

**Data assets stored here:**
- Master IDS (Integrated Diagnostic System) — 397 conditions with symptom weights, breed risk flags, lethality levels, prevalence headwinds
- Breed Bio table — breed-specific genetic risk data used to calculate Breed Surge (B_m)
- Merck Manual reference data — cached periodically from the Merck API (updates X times per year) for specific categories (breeds, symptoms, drugs); used by Level 2 as a post-calculation validation check
- Condition Tags

**The Valtorta Logic Engine — Total Diagnostic Score (TDS) Formula**

The TDS formula is executed by Level 2 against the data assets in Level 3. For each of the 397 conditions in the Master IDS, the engine calculates:

```
TDS = (Symptom_Vector × Age_Factor) + Breed_Surge + Medications_Factor + History_Surge + Acute_Surge − Stability_Penalty + Prevalence_Headwind
```

The scoring components are:

**1. Anatomical Gate (Fg)**
If a condition has a gender requirement that doesn't match the patient, it receives a score of -100 and is excluded. This is a hard gate, not a soft penalty.

**2. Symptom Velocity Engine (Σ W × M)**
Each submitted symptom is matched against the condition's weight in the IDS. Primary symptoms receive a 1.5x multiplier; Secondary symptoms receive 1.0x. Weights can be negative (e.g., -2.0 for Hard Abdomen in Heat Stroke) to enable mutual exclusion logic.

**3. Contextual Surges**
- Age Factor (Af): 1.5x for geriatric conditions if patient age > 8; 0.5x for pediatric if age < 1; 1.0x otherwise.
- Breed Surge (Bm): +1.0 if the patient's breed is in the condition's high-risk breed list (sourced from the Breed Bio table in Level 3).
- Medications Factor: Applied based on current medications in the patient profile. *(Exact weighting and interaction logic TBD — see open questions.)*
- History Surge (Hh): +2.0 if the condition appears in the patient's known medical history.
- Acute Emergency Surge (Ea): +0.5 for crisis-type conditions when duration is Seconds/Minutes.

**4. Sentry Stability Brakes (Ss)**
For high-lethality conditions, each stable sentry marker (Appetite: YES, Energy: YES, Gums: YES, Stool: YES) applies a -2.0 penalty. A dog showing normal appetite and energy is unlikely to be in an acute high-lethality crisis — this is the mechanism that suppresses toxicity and hemorrhage diagnoses when stability markers are positive.

**5. Prevalence Headwinds (Ph)**
A penalty of 0.0, -0.25, or -1.0 applied to rare conditions to prevent statistical outliers from dominating results.

**6. Result Normalization**
All 397 TDS scores are sorted descending. The Top 10 are surfaced.

**The Valtorta Response Output Format**

Results are bucketed into three urgency tiers based on the Sovereign Grip Score (SGS):

| Zone | SGS Range | Action Command | UI Color |
| :---- | :---- | :---- | :---- |
| **CRITICAL STRIKE** | 3.0 + | EMERGENCY: GO NOW | Neon Red |
| **URGENT CONSULT** | 1.5 – 2.9 | CLINICAL REVIEW REQUIRED | Amber Yellow |
| **MONITOR / GUIDANCE** | Below 1.5 | OBSERVE & DOCUMENT | Cobalt Blue |

Each result includes the condition name, SGS score, and a Diagnostic Handshake narrative (LLM-generated by Level 2) explaining why the condition matched.

**Objectives**

- Store and serve the Master IDS, Breed Bio table, Merck Manual reference data, and Condition Tags as queryable assets.
- Allow Level 2 to read from these assets on demand during clinical synthesis.
- Enable clinical weight updates, new condition entries, and breed table changes without requiring code changes.
- Run within a locked VPC perimeter, accessible only to the Level 2 service account.

## Future Capabilities (Post-MVP)

### Multimodal Input Support

While not part of the MVP, the system architecture should be designed to support multimodal inputs (images, videos, PDFs, etc.) in future iterations. The approach:

- Users can upload images or videos of symptoms (e.g., a rash, wound, or physical condition)
- A preprocessing layer extracts clinical features from the media and converts them to symptom text (via model analysis or third-party image recognition API)
- The extracted symptom is fed into the standard mathematical formula as an additional input
- This approach maintains full compatibility with the current deterministic scoring architecture

This design principle allows multimodal features to be added without requiring system redesign or retraining.

## Cross-Cutting Concerns

### Cost & Efficiency Strategy

The architecture prioritizes cost efficiency through a closed-loop design:

- **Deterministic scoring**: The mathematical formula handles the core diagnostic logic without repeated LLM calls. The LLM is used only to generate the Diagnostic Handshake narrative on top of the computed score.
- **Caching**: Results are cached to avoid re-processing identical inputs, reducing token usage across repeated queries.
- **Closed-loop data**: Most data is stored and cached locally in Level 3, minimizing API calls to external services (e.g., Merck Manual is pulled periodically, not queried on-demand).

## Cross-Cutting Concerns

The following issues apply across multiple layers and need to be resolved before the system can be scoped or designed.

### 1. The Master IDS is the product — and we need access to it.

Every other component in this architecture is infrastructure. The actual value proposition of VETALOGIX lives in the 397-condition Master IDS: its symptom weights, breed risk classifications, lethality levels, and prevalence headwinds. The Valtorta Logic Engine algorithm is now defined and understood. The remaining gap is the dataset itself. Until we have the IDS in a usable format and can confirm the data quality, Level 2 (where the scoring runs) and Level 3 (where the data lives) cannot be scoped, estimated, or built. This is the first thing to resolve.

### 2. Regulatory and liability exposure.

The output uses urgency tiers framed as action commands ("EMERGENCY: GO NOW", "CLINICAL REVIEW REQUIRED", "OBSERVE & DOCUMENT"). This is more defensible than a "definitive diagnosis," but AI systems that generate health assessments — even for animals — can attract FDA or USDA scrutiny depending on how they are marketed and used. Before the product is designed, the client needs legal input on how outputs must be framed and disclaimed. This shapes UX copy and disclaimers throughout the system.
